# Usage Guide

This guide provides instructions on how to use the tools in this repository.

## Indexer

The indexer is a tool that generates a code index for the AI agent. To run the indexer, use the following command:

```bash
make index
```

This will generate the index files in the `.ai-agents/index` directory.


## Prerequisites

1. **Claude Code CLI** installed and configured
2. **Go 1.21+** installed
3. **Static analysis tools** (optional but recommended):
   ```bash
   # Install golangci-lint
   curl -sSfL https://raw.githubusercontent.com/golangci/golangci-lint/master/install.sh | sh -s -- -b $(go env GOPATH)/bin

   # Install gosec
   go install github.com/securego/gosec/v2/cmd/gosec@latest

   # Install govulncheck
   go install golang.org/x/vuln/cmd/govulncheck@latest
   ```

## How It Works

### Architecture Overview

The system consists of 5 specialized AI agents coordinated by a workflow orchestrator:

```
User Request
    |
    v
Orchestrator (Router)
    |
    +---> Agent Plan   --> designs architecture, creates plan
    +---> Agent Code   --> generates Go code from plan
    +---> Agent Test   --> generates table-driven tests
    +---> Agent Review --> reviews code quality + security
    +---> Agent Fix    --> debugs and fixes issues
```

### Rules Engine

All agents read the `.rules/` directory before performing any task. These rules enforce consistent Go conventions, architecture, design patterns, security, and testing standards across the entire system.

| Rule File | What It Enforces |
|-----------|------------------|
| `go.md` | Go naming, error handling, context usage, interfaces (consumer-side), concurrency, logging |
| `architecture.md` | Clean Architecture layers, dependency direction (`handler -> service -> repository`), project layout, DI rules |
| `design-patterns.md` | 14 Go-idiomatic patterns (Factory, Options, Repository, Circuit Breaker...), anti-patterns checklist |
| `security.md` | OWASP Top 10 (2025), JWT, SQL injection, crypto, rate limiting, Go-specific security issues |
| `testing.md` | Table-driven tests, gomock, testify, coverage targets per layer, anti coverage-trap rules |

### Context Store

Agents share state through the `.ai-agents/` directory:

```
.ai-agents/
|-- workflow-state.json       # Current state (IDLE, PLANNING, CODING, etc.)
|-- plan.md                   # Generated implementation plan
|-- architecture.md           # Generated architecture diagrams (Mermaid)
|-- tests-plan.md             # Generated test plan
|-- reviews/                  # Review reports (review-1.md, review-2.md, ...)
|-- knowledge/                # Persistent memory across sessions
|   |-- bugs-history.md       # Past bugs, root causes, fixes
|   +-- architecture-decisions.md  # Architecture Decision Records (ADRs)
+-- index/                    # Code index (auto-generated by AST parser)
    |-- symbols.json          # Functions, types, interfaces
    |-- imports.json          # Import graph
    +-- packages.json         # Package structure
```

## Agent Commands

### Full Pipeline (Plan -> Code -> Test -> Review)

```
/agent-full "Add JWT authentication with refresh tokens"
```

This runs the complete workflow:
1. **Agent Plan** designs architecture and creates plan
2. **Agent Code** generates Go source code following the plan
3. **Agent Test** generates table-driven tests with mocks
4. **Agent Review** checks quality + security (OWASP Top 10)
5. If issues found: **Agent Fix** repairs, then re-reviews (max 3 loops)
6. If 3 loops exceeded: orchestrator stops and asks you

### Individual Agents

#### Agent Plan - Design & Architecture
```
/agent-plan "Add user management with CRUD endpoints"
```

**What it does:**
- Reads all `.rules/` files to understand project conventions
- Reads existing code index (`.ai-agents/index/`) to understand current codebase
- Asks clarifying questions if requirements are unclear
- Designs Go project layout following Clean Architecture (`cmd/`, `internal/`, `pkg/`)
- Selects appropriate design patterns from the 14 available patterns
- Creates Mermaid architecture diagrams
- Generates test plan with coverage targets per layer

**Output files:**
- `.ai-agents/plan.md` - Step-by-step implementation plan with file list, dependencies, task order
- `.ai-agents/architecture.md` - Layer diagrams, dependency flow, component interactions
- `.ai-agents/tests-plan.md` - Test cases organized by layer with expected coverage

**Example plan.md structure:**
```markdown
## Feature: User Management CRUD
### Task List:
1. [ ] Define domain entities (User, UserRole)
2. [ ] Create service interface + implementation
3. [ ] Create repository interface at service layer (consumer-side)
4. [ ] Implement PostgreSQL repository
5. [ ] Create HTTP handlers with input validation
6. [ ] Wire dependencies in main.go

### Design Patterns Used:
- Repository Pattern (data access abstraction)
- Functional Options (service configuration)
- Factory (entity creation with validation)
```

#### Agent Code - Generate Code
```
/agent-code
```

**What it does:**
- Reads plan from `.ai-agents/plan.md`
- **Validates architecture** before generating code (drift prevention)
- Generates Go code following SOLID + Clean Architecture + Design Patterns
- Places interfaces at consumer side (Go idiom: "Accept interfaces, return structs")
- Applies secure coding checklist (input validation, parameterized queries, proper error wrapping)
- Runs pre-completion checklist (12 items including `go vet`, `goimports`, no hardcoded secrets)

**Architecture validation checks:**
- Import direction: `handler -> service -> repository` (never reverse)
- No circular dependencies between packages
- Domain layer has zero external imports
- Interfaces defined at consumer, not provider

**Reads:** `.rules/*`, `.ai-agents/plan.md`, `.ai-agents/architecture.md`

#### Agent Test - Generate Tests
```
/agent-test
```

**What it does:**
- Reads `.ai-agents/tests-plan.md` and existing source code
- Generates **table-driven tests** with `testify/assert` and `testify/require`
- Creates mocks using `gomock` (generates mock files with `mockgen`)
- Covers edge cases: nil input, empty collections, boundary values, context cancellation, timeout
- Runs `go test -race` to detect race conditions
- Checks coverage against targets

**Anti coverage-trap rules (FORBIDDEN):**
```go
// FORBIDDEN - test without assertion
func TestSomething(t *testing.T) {
    result, _ := DoSomething()  // no assert = coverage trap
}

// REQUIRED - every test case must have assert or require
func TestSomething(t *testing.T) {
    result, err := DoSomething()
    require.NoError(t, err)
    assert.Equal(t, expected, result)
}
```

**Coverage targets:**
| Layer | Target |
|-------|--------|
| domain/ | 90% |
| service/ | 85% |
| handler/ | 80% |
| repository/ | 70% |
| Critical paths (auth, payment) | 95% |

#### Agent Fix - Debug & Fix Bugs
```
/agent-fix "Error: nil pointer dereference at internal/service/user.go:42"
```

**What it does:**
1. **Analyze** - Reads error message, stack trace, identifies affected file and line
2. **Investigate** - Uses `git blame` and `git log -p` to understand history and context
3. **Go-specific checks** - goroutine leaks, nil interface, race conditions, swallowed errors
4. **Test first** - Writes regression test BEFORE applying the fix
5. **Fix** - Applies minimal, targeted fix
6. **Verify** - Runs `go test -race` to confirm fix and no regressions
7. **Record** - Saves bug details in `.ai-agents/knowledge/bugs-history.md`

**Rollback strategy:**
- Creates temp branch `agent-wip-fix-<description>`
- If fix causes new failures: `git reset --hard` and reports to user

#### Agent Review - Code Review & Security Audit
```
/agent-review
```

**3-layer review pipeline:**

| Layer | Tool | What It Checks |
|-------|------|----------------|
| 1. Static Analysis | golangci-lint, gosec, govulncheck | Code quality, security patterns, known vulnerabilities |
| 2. AI Review | Claude | SOLID compliance, design pattern usage, architecture adherence, business logic correctness |
| 3. Security Review | OWASP checklist | Top 10 vulnerabilities + Go-specific: JSON injection, goroutine leaks, context timeout, GORM SQL injection, race conditions, nil interface |

**Design patterns review includes:**
- Pattern compliance check (are selected patterns used correctly?)
- Anti-pattern detection (god struct, circular deps, global state, interface pollution)
- Pattern misuse detection (unnecessary abstraction, wrong pattern for the problem)

**Security-only review:**
```
/agent-review --security-only
```
Skips Layer 2 (AI code review) and only runs static analysis + OWASP security checklist.

**Severity levels:**
| Level | Meaning | Action |
|-------|---------|--------|
| CRITICAL | Security vulnerability, data loss risk | Must fix before merge |
| HIGH | Bug, significant design flaw | Must fix before merge |
| MEDIUM | Code smell, minor design issue | Should fix |
| LOW | Style, naming, minor improvement | Optional |

**Output:** `.ai-agents/reviews/review-N.md` (auto-versioned)

## Workflow State Machine

```
IDLE --> PLANNING --> CODING --> TESTING --> REVIEWING --> DONE
                                               |
                                               v (issues found)
                                             FIXING --> REVIEWING
                                                          |
                                                          v (max 3 loops)
                                                        ESCALATE (asks user)
```

- State tracked in `.ai-agents/workflow-state.json`
- Agents work on a temp branch: `agent-wip-<feature>`
- **Never auto-commits or pushes** - you decide when to commit
- If review-fix loops exceed 3, the orchestrator stops and asks you

### Workflow State File

```json
{
  "state": "IDLE",
  "feature": "",
  "branch": "",
  "loop_count": 0,
  "max_loops": 3,
  "current_agent": "",
  "last_updated": ""
}
```

States: `IDLE`, `PLANNING`, `CODING`, `TESTING`, `REVIEWING`, `FIXING`, `DONE`, `ESCALATED`

## Project Rules - Customization Guide

Rules in `.rules/` are read by ALL agents before any task. You can customize them to match your project.

### Customizing Go Conventions (`.rules/go.md`)

```markdown
# Example: Change your project's import grouping order
## Import Order
1. Standard library
2. Your organization packages (e.g., github.com/yourorg/*)
3. Third-party packages
4. Internal packages
```

### Customizing Architecture (`.rules/architecture.md`)

```markdown
# Example: Add a new layer for your project
## Project Structure
your-project/
|-- cmd/api/          # Entry point
|-- internal/
|   |-- domain/       # Entities, value objects (zero external imports)
|   |-- service/      # Business logic + interfaces for dependencies
|   |-- repository/   # Data access implementations
|   |-- handler/      # HTTP/gRPC handlers
|   |-- worker/       # Background job processors (custom layer)
|   +-- infra/        # External service clients
```

### Customizing Design Patterns (`.rules/design-patterns.md`)

```markdown
# Example: Make Circuit Breaker optional for internal services
## Circuit Breaker
- WHEN: External API calls, third-party services
- NOT NEEDED: Internal service-to-service calls within trusted network
```

### Customizing Security (`.rules/security.md`)

```markdown
# Example: Add project-specific security rules
## Custom Rules
- All API endpoints must require authentication except /health and /metrics
- PII fields must be encrypted at rest using AES-256-GCM
- All database queries must use parameterized statements (no string concatenation)
```

### Customizing Testing (`.rules/testing.md`)

```markdown
# Example: Adjust coverage targets for your project
## Coverage Policy
| Layer | Target |
|-------|--------|
| domain/ | 95% |    # Increased for critical business logic
| service/ | 90% |   # Increased
| handler/ | 75% |   # Relaxed for thin handlers
| repository/ | 60% | # Relaxed for simple CRUD
```

## Code Index

The code index helps agents understand your codebase without reading every file. It is generated by a **Go AST parser** (`tools/indexer/main.go`), NOT by the LLM, to prevent hallucination.

```bash
# Generate/update the code index
make update-index
```

This creates:
- `.ai-agents/index/symbols.json` - All functions, methods, types, interfaces with file:line locations
- `.ai-agents/index/imports.json` - Import graph showing package dependencies
- `.ai-agents/index/packages.json` - Package structure with doc comments

**When to run:** After significant code changes, before running agents.

**Note:** Agents call `make update-index` automatically when needed. The index files are gitignored (regenerated on demand).

### How agents use the index

1. **Agent Plan** reads `packages.json` to understand existing structure before designing new features
2. **Agent Code** reads `symbols.json` to find existing types/interfaces to implement or extend
3. **Agent Fix** reads `imports.json` to trace dependency chains when debugging
4. **Agent Review** reads `symbols.json` to verify architecture rules (no forbidden imports)

## Knowledge Memory

Agents learn from experience via persistent knowledge files in `.ai-agents/knowledge/`:

### bugs-history.md

Written by **Agent Fix** after each successful fix. Helps agents avoid repeating mistakes.

```markdown
### [2024-03-15] Bug: nil pointer in OrderService.PlaceOrder
- **File:** internal/service/order.go:87
- **Root cause:** Repository field not injected in constructor
- **Fix:** Added required repo parameter to NewOrderService()
- **Regression test:** internal/service/order_test.go:TestPlaceOrder_NilRepo
- **Prevention:** Always validate required dependencies in constructors
```

### architecture-decisions.md

Written by **Agent Plan** during design. Records why certain decisions were made.

```markdown
### ADR-001: Use PostgreSQL over MongoDB for user data
- **Date:** 2024-03-15
- **Status:** Accepted
- **Context:** User data has strong relational structure with many joins
- **Decision:** PostgreSQL with GORM
- **Alternatives:** MongoDB (rejected: poor join support), SQLite (rejected: not production-ready)
- **Consequences:** Need migrations strategy, GORM learning curve
```

## Safety Guarantees

| Safety Feature | Description |
|----------------|-------------|
| No auto-commit | Agents never commit or push code without your explicit request |
| No auto-merge | Work stays on temp branch (`agent-wip-*`), you decide to merge |
| Loop limit | Review-Fix loop capped at 3 iterations, then escalates to you |
| Rollback | If fix causes new failures, agents revert (`git reset --hard`) and report |
| Architecture validation | Agent Code validates against `architecture.md` before generating code |
| Drift prevention | Generated code is checked against the original plan to prevent deviation |
| AST-based index | Code index generated by Go parser, not LLM, to prevent hallucination |

## Troubleshooting

### Agent doesn't read the rules
Make sure `CLAUDE.md` exists in the project root. Claude Code reads it automatically at session start. Verify it contains references to `.rules/` and `.ai-agents/`.

### Code index is empty
Run `make update-index` manually. The indexer needs Go source files (`.go`) to parse. It skips `vendor/`, `testdata/`, hidden directories, and `_test.go` files.

### Static analysis tools not found
Install them:
```bash
go install github.com/golangci/golangci-lint/cmd/golangci-lint@latest
go install github.com/securego/gosec/v2/cmd/gosec@latest
go install golang.org/x/vuln/cmd/govulncheck@latest
```
If tools are not in PATH, add `$(go env GOPATH)/bin` to your PATH:
```bash
export PATH=$PATH:$(go env GOPATH)/bin
```

### Agent generates code that violates architecture
Check `.rules/architecture.md` for the dependency rules. The agent validates against these rules, but if the plan itself has violations, fix the plan first with `/agent-plan`. You can also edit `.ai-agents/plan.md` directly before running `/agent-code`.

### Review-Fix loop hits max (3 iterations)
The orchestrator will stop and ask you. Options:
1. Fix manually based on the review report in `.ai-agents/reviews/`
2. Adjust the requirements and re-run `/agent-plan`
3. Relax rules in `.rules/` if they're too strict for your case
4. Increase `max_loops` in `.ai-agents/workflow-state.json` (not recommended)

### Agent produces different results each run
This is expected behavior for LLM-based agents. To improve consistency:
- Make `.rules/` files more specific with concrete examples
- Provide detailed requirements in your prompt
- Use the code index (`make update-index`) so agents understand existing patterns

### Tests fail with "mockgen not found"
Install mockgen:
```bash
go install go.uber.org/mock/mockgen@latest
```

### Integration tests fail
Integration tests use `testcontainers` which requires Docker. Make sure Docker is running:
```bash
docker info  # Should show Docker daemon info
make test-integration
```
